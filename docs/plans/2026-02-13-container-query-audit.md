# Container Query Audit (Issue #220)

## Scope

- Files scanned: `src/**/*.astro`, `src/**/*.tsx`, `src/**/*.jsx`
- Pattern scanned: `sm:|md:|lg:|xl:`
- Goal: classify all responsive breakpoint hits for container-query migration planning

## Method

1. Static inventory created with:
   - `rg -n "sm:|md:|lg:|xl:" src --glob '*.{astro,tsx,jsx}' > /tmp/container-query-inventory.txt`
2. Each hit categorized using `design-system/05-responsive.md` rubric:
   - `MIGRATE_TO_CONTAINER`
   - `KEEP_VIEWPORT_CHROME`
   - `NEEDS_DECISION`
3. Rationale assigned per hit based on responsive role (component layout vs chrome vs ambiguous/token-map/marketing)

## Mobile UX Findings

- Touch targets: pagination/filter/action controls generally enforce `min-h-[44px]` and `min-w-[44px]`; preserve this requirement during migration.
- Navigation and chrome: sidebar/header/mobile-nav visibility uses viewport breakpoints and should stay viewport-driven (`KEEP_VIEWPORT_CHROME`).
- Form usability: multiple forms still use viewport row switches (`sm:flex-row`, `md:grid-cols-*`), which can collapse poorly in narrow side panels; prioritize these for container-query conversion.
- Data-heavy layouts: table/card toggles remain mixed (some already migrated with `@3xl`, others still `md:`/`lg:`); remaining viewport toggles should be normalized for consistent behavior in constrained containers.

## Out of Scope

- Behavioral/UI refactors unrelated to responsive breakpoint semantics
- Any code changes to production components (audit-only deliverable)
